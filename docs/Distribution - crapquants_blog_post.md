## The Problem Nobody Talks About at Demo Day

Vibe coding has won. The debate is over. Karpathy coined the term in February 2025, and by mid-2026, 84% of developers use AI coding tools daily. Cursor hit $2 billion ARR. GitHub reports 55% faster task completion with Copilot. Microsoft's CTO predicted 95% of all code will be AI-generated by 2030.

But beneath the speed, something is quietly going wrong.

**The data is brutal:**

- GitClear analyzed 211 million lines of code and found an **8-fold increase in duplicate code blocks** during 2024. Copy-paste now exceeds code reuse for the first time in history. The DRY principle — Don't Repeat Yourself — is dying.
- CodeRabbit found AI-coauthored pull requests had **2.74× more security vulnerabilities** than human-written ones. Hardcoded API keys. Plain-text passwords. APIs fabricated from outdated training data.
- Code churn — code rewritten within two weeks of being written — rose from 3.1% in 2020 to 5.7% in 2024. More code, faster, is not the same as more value, faster.

And here's the finding that should keep you up at night: a senior engineer at Cursor itself uncovered in March 2026 that the AI builds for now, not for later. It has no stake in the maintainability of what it generates.

**This is the "three-month black box" problem.** You vibe-code a project. Three months later, nobody — including you — understands the AI-generated structure well enough to change it safely. The project isn't dead because it doesn't work. It's dead because nobody can touch it.

## Why Standard Tools Don't Catch This

You might think: "I run linters. I have CI. I write tests."

Good. But those tools answer the wrong question.

- **Linters** tell you about style violations. They catch missing docstrings. They don't tell you that your 47-line payment validation function has 12 branching paths and zero test coverage.
- **Coverage tools** tell you what lines were executed during tests. They don't tell you whether the *important* paths through complex code are actually tested. You can have 80% coverage and still have every dangerous function completely untested.
- **Type checkers** prevent category errors. They can't detect that an LLM duplicated business logic across three files with slightly different edge-case handling.

None of these tools answer the fundamental maintainability question: **where in this codebase is it dangerous to make changes?**

That's the gap. And it's exactly the gap that grows fastest when LLMs write your code, because LLMs optimize for "it works right now" — not for "someone can safely change this later."

## What CRAPQuants Actually Does

CRAPQuants is a Python-native static analysis tool that computes the CRAP metric — **Change Risk Anti-Patterns** — for every function in your codebase. The formula is simple:

```
CRAP(m) = CC² × (1 − coverage/100)³ + CC
```

Where CC is cyclomatic complexity (how many branching paths exist) and coverage is how much of that function your tests actually exercise.

The intuition is dead simple: **a function that's both complex and poorly tested is dangerous to change.** CRAP captures that in a single number. Score above 30? That function is CRAPpy — high risk, prioritize it.

The metric was created by Alberto Savoia and Bob Evans at Google in 2007. It was originally implemented for Java (Crap4J, now abandoned) and recently brought to Rust (cargo-crap). CRAPQuants is the Python implementation — and it goes far beyond just the number.

### What Makes CRAPQuants Different: Six Books, Not Just One Formula

Here's where it gets interesting. A CRAP score tells you *where* the risk is. It doesn't tell you *what kind* of risk it is, or *what to do about it.*

CRAPQuants enriches every function's analysis with diagnostic tags derived from **six foundational software engineering books**:

**1. Michael Feathers — Working Effectively with Legacy Code**
Tags like `MONSTER_SNARLED` (function too tangled to test), `EDIT_AND_PRAY` (no tests, you're gambling), `CHARACTERIZATION_NEEDED` (write tests that document current behavior before touching anything). Feathers' core insight: code without tests *is* legacy code, regardless of when it was written. Every function an LLM generates without corresponding tests is instant legacy.

**2. John Ousterhout — A Philosophy of Software Design**
Tags like `SHALLOW_MODULE` (interface as complex as implementation — the opposite of good design), `NONOBVIOUS` (complexity hidden behind simple-looking code), `DEEP_MODULE` (what good design looks like). Ousterhout's warning maps directly to AI-generated code: LLMs create functions that *look* clean but hide complexity in the wrong places.

**3. Hunt & Thomas — The Pragmatic Programmer**
Tags like `BROKEN_WINDOW` (one bad function signals the whole area is neglected), `COINCIDENCE_CODE` (it works, but nobody knows why — classic LLM output), `NO_SAFETY_NET` (complex code with no tests). The Pragmatic Health Score (PHS, 0–100) gives your codebase a single sustainability number.

**4. Martin Fowler — Refactoring**
Tags like `LONG_METHOD`, `LONG_PARAM_LIST`, `CONDITIONAL_CHAIN`, `FEATURE_ENVY`. These are the classic code smells — and LLMs produce them at scale because they don't refactor. They generate. Every time an AI adds another `if` branch instead of extracting a function, Fowler's patterns detect it.

**5. Adam Tornhill — Your Code as a Crime Scene**
This is the behavioral layer — powered by real git data. Tags like `HOTSPOT_ACTIVE` (frequently changed AND complex — the most dangerous combination), `KNOWLEDGE_SILO` (only one person has ever touched this code), `TREND_DETERIORATING` (complexity increasing over time). Tornhill's insight: the files that change the most AND have the highest complexity are where bugs cluster. When an LLM keeps modifying the same complex file, CRAPQuants flags it.

**6. Neal Ford — Building Evolutionary Architectures**
Fitness function tags like `FITNESS_CRAP_EXCEEDED`, `FITNESS_COGC_EXCEEDED`. Ford's argument: architecture must be testable with automated fitness functions. CRAPQuants *is* a fitness function — it gates your CI pipeline against architectural decay.

Plus **SonarSource's Cognitive Complexity** (how hard a function is for a *human brain* to understand) and the **ABC metric** (Assignments, Branches, Conditions) for a complete picture.

### Beyond Metrics: Git Behavioral Analysis

CRAPQuants doesn't just look at code in isolation. At the `deep` analysis level, it examines your git history:

- **Hotspot scoring**: churn × CRAP. The file that changes every sprint AND has the worst scores? That's your number one priority.
- **Change coupling**: files that always change together — a sign of hidden dependencies that LLMs love to create.
- **Truck factor**: how many people need to get hit by a truck before nobody understands this code? In a vibe-coded project with one developer, this number is often exactly one.
- **Complexity trends**: is a function getting better or worse over time? `DETERIORATING` / `STABLE` / `IMPROVING` — tracked automatically.

### Security Scanning Built In

At the `full` analysis level, CRAPQuants includes two-tier security scanning:

- **Tier 1 (always on)**: AST-based detection of eval/exec usage (CWE-95), pickle deserialization (CWE-502), hardcoded secrets (CWE-798), shell injection (CWE-78), and unsafe YAML loading (CWE-502). These are the exact vulnerability patterns LLMs inject most frequently.
- **Tier 2 (optional)**: Semgrep integration for comprehensive SAST.

This matters because 40–62% of AI-generated code contains security vulnerabilities. CRAPQuants catches the most common ones as part of the same analysis run.

## Why Vibe Coders Specifically Need This

If you're writing every line by hand, you already have an intuitive sense of which parts of your code are fragile. You wrote them. You remember the shortcuts. You know where the dragons are.

**Vibe coders don't have this intuition.** When an LLM writes a function, you might read it, you might not. Either way, you didn't struggle through the design decisions. You don't have the muscle memory of "this was hard to write, so it'll be hard to change." The complexity is invisible to you.

CRAPQuants makes it visible.

Here's the workflow that actually works:

```bash
# 1. Let the LLM generate code
# 2. Run your tests with coverage
pytest --cov=src --cov-report=json

# 3. Run CRAPQuants
crapquants analyze ./src --coverage coverage.json

# 4. If anything scores above 30, go back to the LLM:
#    "This function has a CRAP score of 156.
#     Reduce complexity or add tests for the critical paths."
```

Four commands. Thirty seconds. You now have visibility into what the AI actually generated.

For teams, baseline regression detection is even more powerful:

```bash
# Save baseline on main
crapquants analyze ./src -c coverage.json --save-baseline data/baseline.json

# On every PR — exit code 1 if anything got worse
crapquants analyze ./src -c coverage.json --baseline data/baseline.json
```

Your codebase may have existing problems. That's fine. This rule simply says: **new changes cannot make things worse.** Pre-existing CRAPpy functions don't block your CI. Only regressions do.
